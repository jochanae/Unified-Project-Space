# Axiom — Strategic Thinking Partner

A unified development environment combining the Axiom frontend (atlas-idk) and backend (Axiom-Atlas) in one place.

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 8080, path: /api)
- `pnpm --filter @workspace/atlas-frontend run dev` — run the frontend (port 22883, path: /)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string (currently Supabase)

## Stack

- pnpm workspaces, Node.js 24, TypeScript 5.9
- Frontend: React + Vite, Tailwind CSS v4, Radix UI, Supabase auth
- API: Express 5
- DB: Supabase PostgreSQL
- Validation: Zod (`zod/v4`), `drizzle-zod`
- API codegen: Orval (from OpenAPI spec)
- Build: esbuild (CJS bundle)

## Where things live

- `artifacts/atlas-frontend/` — the full Axiom frontend (sourced from github.com/jochanae/atlas-idk)
- `artifacts/atlas-frontend/src/_workspace/api-client-react/` — the atlas-idk's own API client with Axiom-Atlas backend hooks (aliased as `@workspace/api-client-react` in vite.config.ts)
- `artifacts/api-server/` — Express backend (template, pending Axiom-Atlas backend migration)
- `lib/api-spec/openapi.yaml` — OpenAPI spec (for the Replit monorepo backend)
- `lib/db/src/schema/index.ts` — DB schema (⚠️ DO NOT TOUCH lib/db/src/index.ts)

## Architecture decisions

- The atlas-frontend carries its own copy of the API client (`src/_workspace/api-client-react/`) which contains the generated hooks matching the Axiom-Atlas backend's OpenAPI spec. This local copy is aliased in vite.config.ts instead of the monorepo's `lib/api-client-react`.
- `VITE_API_URL` points to the live Cloud Run backend at `https://axiom-atlas-689827072865.us-east1.run.app` — this means the frontend talks to the production backend while running locally.
- The frontend uses Supabase for auth (`VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY` set as env vars).
- The `lib/db/src/index.ts` file must NEVER be modified — caused a 3-day outage previously.

## Product

Axiom is a strategic thinking partner — a workspace where ideas hold their shape long enough to become real. It includes a workspace, project management, decision ledger, parking lot, sessions, vault, and AI chat capabilities.

## User preferences

- Never touch `lib/db/src/index.ts`
- No visual deviations from the original atlas-idk frontend
- `workspace.tsx` is 400KB+ — handle with care (use bash cp, never read into context)

## Gotchas

- `workspace.tsx` is 400KB+ — never read into context, use `cp` or line-range reads only
- `VITE_API_URL` must point to the backend — currently set to Cloud Run URL; change if running the Axiom-Atlas backend locally
- The frontend uses Bun in the original repo but runs fine with pnpm/vite here
- The `src/_workspace/api-client-react/` is NOT auto-generated by the Replit codegen — it was sourced from the atlas-idk repo and must be updated manually if the Axiom-Atlas OpenAPI spec changes
- Auth works via Supabase (cookie + localStorage bearer token); 401s on load are expected before login
